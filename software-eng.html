Software Life Cycle: Development Phase
  1. Requirements Analysis
  2. Design
  3. Implementation
  4. Testing
  1. Specify what services the proposed system will provide, identify any conditions on those services, and define how the outside world will interact with the system.
  2. Create a plan for the construction of the proposed system.
  3. Write programs, create data files, and develop databases.
  4. Debug programs and confirm that the final software product is compatible with software requirements.
Modular Implementation The division of software into manageable units, generically called modules, each of which deals with only a part of the software's overall responsibility.
Unified Modeling Language (UML) A modern collection of tools developed with the object-oriented paradigm in mind.
Use Case Interactions between a system and its users.
Use Case Diagram A diagram which depicts a proposed system as a large rectangle in which interactions (or use cases) between the system and its users are represented as ovals and users of the system (or actors) are represented as stick figures.
Data Relationships:
  1. one-to-one
  2. one-to-many
  3. many-to-many
  1. A one-to-one relationship is exemplified by the association between patients and occupied private rooms in that each patient is associated with only one room and each room is associated with only one patient.
  2. A one-to-many relationship is exemplified by the association between physicians and patients in that one physician is associated with many patients and each patient is associated with one physician.
  3. A many-to-many relationship would occur if we included consulting physicians in the physician-patient relationship. Then each physician could be associated with many patients and each patient could be associated with many physicians.
Design Pattern A predeveloped model for solving a recurring problem in software design.
Software Quality Assurance (SQA) Groups charged with overseeing and enforcing the quality control systems adopted by the organization.
Documentation:
  1. glass box testing
  2. black box testing
  3. beta testing
  4. alpha testing
  1. Testing where the software tester is aware of the interior structure of the software and uses this knowledge when designing the test.
  2. Testing that does not rely on knowledge of the software's interior composition.
  3. A form of black box testing where a preliminary version of the software is given to a segment of the intended audience in order to determine how it performs in real-life situations.
  4. A form of developer-side beta testing.
Documentation:
  1. user documentation
  2. system documentation
  3. technical documentation
  1. Documentation that explains the features of a software system and describes how to use it.
  2. Documentation that describes a software system's internal composition so it can be maintained later in life.
  3. Documentation that describes how a software system should be installed and serviced.
Intellectual Property Law:
  1. software license
  2. software disclaimer
  3. best defenses against software liability claims
  1. A legal agreement between the owner and user of a software product that grants the user certain permissions without transferring ownership.
  2. Statements in software licenses which outline the limitations of developer liability.
  3. Applying sound software engineering principles during the software's development, using a level of care compatible with the software's application, and producing / maintaining records that validate these endeavors.
